## 关键决策与理由（防止“吃书”）

- 决策A：第一版使用 SVG DOM，不使用 Canvas。（原因：计划书要求路径、标定点、参考线都以矢量对象呈现，SVG 更容易与 G-code 路径模型对应。）
- 决策B：先实现测试路径与假字体，再接入真实中文笔画字库。（原因：先验证坐标和 G-code 闭环，避免字体数据源成为早期不确定因素。）
- 决策C：不实现串口发送器，只导出 `.gcode` 文件。（原因：MVP 明确要求用户通过外部 sender 执行。）
- 决策D：项目状态使用 Zustand + localStorage 持久化。（原因：第 1 步要求能保存基础 project state，Zustand API 简单，适合 MVP 阶段快速验证。）
- 决策E：画布缩放和平移只改变 SVG `viewBox`，不改变纸面对象坐标。（原因：纸面对象必须一直保持毫米坐标，避免 UI 缩放影响后续 G-code 导出。）
- 决策F：测试图案先转换为 `PolylinePath[]`，G-code 阶段不从 SVG DOM 反推路径。（原因：导出逻辑必须依赖项目数据，而不是依赖浏览器渲染结果。）
- 决策G：基础 G-code 导出先把当前纸面路径按机器路径处理，后续第 6 步再接入机器参考线旋转映射。（原因：计划书实现顺序要求先跑通测试路径导出，再处理照片/机器参考线标定。）
- 决策H：第 5 步先实现 Homography 数学核心与点选状态链路，再增强真实照片透视矫正渲染。（原因：矩阵正确性是后续机器参考线和纸面校正的基础。）
- 决策I：标定中使用照片原始 px viewBox，标定完成后再回到纸面 mm viewBox。（原因：计划书要求四角点是照片坐标，不能把照片点误保存成纸面毫米坐标。）
- 决策J：机器参考线在完成四角标定后的纸面视图中点选，但内部会反算回照片 px 保存。（原因：用户看的是校正纸面更直观，数据结构仍保持计划书要求的照片坐标来源。）
- 决策K：归零提示作为独立高亮区块，而不是只放在导出前检查列表里。（原因：MVP 不控制串口，用户必须在外部 sender 执行 `G92 X0 Y0` 才能保证坐标原点正确。）
- 决策L：Z 标定面板只生成可复制 G-code，不做串口发送。（原因：MVP 明确不实现串口通信，用户需要用外部 sender 执行测试。）
- 决策M：FakeStrokeFontProvider 只用于验证文本到单线路径的闭环，不追求字形美观。（原因：真实中文笔画字库放到第 10 步单独接入，避免影响前 9 步坐标/G-code 主链路。）
- 决策N：第一种中文 provider 使用内置低保真单线字形覆盖验收词，而不是立即接入大型第三方中文字库。（原因：MVP 验收重点是单线 stroke 路径和坐标/G-code 闭环，字形美观可以在后续字体效果优化阶段迭代。）
- 决策O：机器 Z 方向和 Z 标定 Z 方向保持同步。（原因：同一台机器不应在“机器设置”和“Z 标定流程”中出现互相矛盾的 Z 正方向。）
- 决策P：README 面向 GitHub 展示和用户上手，不重复 `计划书.md` 的全部技术细节。（原因：公开首页需要让访问者快速理解项目用途、启动方式、部署方式和实机验收流程；详细设计仍以 `计划书.md`、`PLANS.md`、`PROGRESS.md` 为准。）
- 决策Q：UI 回到工具型桌面布局，不继续使用三列长表单页面。（原因：真实标定需要一屏看到顶栏工具、主画布、侧栏步骤和底栏状态，不能让机器/Z/导出信息掉到页面滚动下方。）
- 决策R：照片本体存 IndexedDB Blob，localStorage 只保存项目 JSON 和 imageId。（原因：base64 data URL 会膨胀并触发 localStorage 配额错误，大图必须使用浏览器二进制存储。）
- 决策S：Web Serial 只先服务 Z 标定发送，不替代完整 sender。（原因：当前目标是解决 Z 轴实机校准；完整 sender 需要队列、回读、暂停、急停等更大范围设计。）

## 常见坑 / 复现方法

- 坑1：不要假设机器有 home；MVP 依赖用户把笔头移动到纸张左上角后执行 `G92 X0 Y0`。
- 坑2：Z 标定不是自动调平；没有传感器时只能生成点粗找和线细调 G-code，让用户目测选择可用落笔高度。
- 坑3：TypeScript 6 对 CSS side-effect import 更严格，必须保留 `src/vite-env.d.ts` 中的 Vite client 类型引用。
- 坑4：不要给页面写死桌面最小宽度；Codex 内置浏览器和小屏验证会暴露横向溢出，当前通过响应式单列解决。
- 坑5：拖动画布对象会产生浮点数，当前统一保留 0.01mm，并限制对象不拖出纸面。
- 坑6：Vitest/jsdom 不能真实下载文件，导出函数在测试模式只验证生成状态，不触发 `<a>` 下载点击。
- 坑7：SVG 本身不能直接做通用透视校正背景，当前用一次性 Canvas 重采样生成校正 PNG，再作为 SVG 背景显示。
- 坑8：机器 Y 轴参考线要旋转得到机器 X 轴；当前约定纸面 Y 向下时，Y 方向线段 `(0, 1)` 旋转后得到 X 方向 `(1, 0)`。
- 坑9：Z “下降一步”的正负取决于 `zPositiveDirection`；当前封装在 `stepZTowardPaper`，避免 UI 里写死 Z 变小。
- 坑10：软件侧测试只能证明 G-code 生成和坐标映射逻辑，不能替代真实机器验收；最终还要确认外部 sender 的坐标模式、单位模式和用户是否已执行 `G92 X0 Y0`。
- 坑11：内置中文 provider 只是 MVP 验收字形，覆盖“你好”“实验报告”；后续要写更多中文，需要扩展字形数据或接入更完整的 stroke 字库。
- 坑12：GitHub Pages 如果部署在仓库子路径，可能需要配置 Vite 的 `base`；Cloudflare Pages、Vercel、Netlify 这类根路径静态部署通常只需要 `npm run build` 和 `dist/`。
- 坑13：HEIC/HEIF 即使能被文件选择器选中，也不代表 Chromium 能解码；当前会显示明确错误，后续若要原生支持需要接入转换器。
- 坑14：Web Serial 必须由用户手动选择串口，且同一串口不能同时被外部 sender 占用。
- 坑15：本轮 Codex in-app browser 自动化读取 DOM 连续超时；命令侧 `typecheck`、`test`、`build` 和本地 HTTP 200 已通过，最终 UI 观感和串口授权弹窗需要用户当前浏览器实操确认。
