# PaperPlotter Agent 工作约定

## 回答与协作

- 始终使用简体中文与用户沟通。
- 面向编程初学者解释实现取舍，尽量说明“为什么这样做”和“如何验证”。
- 复杂任务先维护 `PLANS.md`，阶段完成后同步 `PROGRESS.md`。

## 开发原则

- 优先实现可运行、可验证的最小闭环，避免一开始扩展成完整矢量编辑器或串口控制器。
- 代码以 TypeScript 类型、纯函数和可测试模块为核心，UI 只负责调用和展示状态。
- 文件头与关键函数必须写中文说明，解释模块职责、输入、输出、边界和实现取舍。
- 注释要服务理解，不堆砌无意义复述；复杂公式、坐标变换、G-code 安全检查必须详细说明。

## 版本控制

- 每次完成一组文件修改后立即 `git add` 并 `git commit`。
- 提交信息采用中文 Conventional Commits，例如 `docs: 初始化项目执行计划`。
- 不回滚用户已有文件，遇到不相关改动只忽略。

## 验收要求

- 每个阶段都要至少通过类型检查、单元测试或浏览器验证中的一种。
- 关键算法必须覆盖正常、边界和异常情况。
- MVP 最终验收以 `计划书.md` 的闭环为准：照片标定、纸面对象、G-code 导出、G92 归零提示、单点 Z 标定和假字体路径输出。
