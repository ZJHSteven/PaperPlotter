# 写字机纸面排版与 G-code 生成工具 MVP 开发计划书

## 0. 项目定位

本项目不是完整矢量编辑器，也不是完整写字机控制软件，而是一个面向写字机的 **纸面排版与 G-code 生成工具**。

核心目标是：

用户导入真实纸张照片，通过手动标定纸张四角和机器参考线，在软件中得到一个与真实纸张对应的纸面坐标系；用户可以在纸面上放置文本或路径对象，最后导出可由现有写字机下位机执行的 G-code 文件。

第一版只负责：

* 纸张照片标定；
* 纸面坐标建模；
* 简单对象排版；
* 测试路径预览；
* G-code 文件导出；
* 单点 Z 落笔高度标定流程。

第一版不负责：

* 串口通信；
* 自动 home；
* 真正自动调平；
* 完整矢量编辑；
* 商业字库解包；
* 多机器高级管理；
* 复杂字体市场。

---

## 1. MVP 核心目标

MVP 要验证的最小闭环是：

```text
夹好纸 → 拍照导入 → 标定纸张四角 → 标定机器参考线
→ 手动把笔头移动到纸张左上角 → 外部 sender 执行 G92 X0 Y0
→ 软件中放置测试路径或文本对象 → 导出 G-code → 机器按预览位置写到纸上
```

最关键的验收标准是：

```text
纸张即使没有夹正，只要标定正确，
软件预览中相对纸张的位置，应该尽量对应真实纸张上的位置。
```

---

## 2. MVP 范围

### 2.1 必须实现

1. 纸张配置

   * 支持 A4、A5、B5；
   * 支持自定义宽度和高度；
   * 支持横向 / 纵向。

2. 照片导入

   * 用户导入纸张照片；
   * 照片作为标定背景；
   * 支持在照片上点选四个纸角。

3. 纸张四角标定

   * 用户依次点选：

     * 左上角；
     * 右上角；
     * 右下角；
     * 左下角。
   * 软件根据纸张真实尺寸建立纸面坐标系。
   * UI 中显示校正后的纸面背景。

4. 机器参考线标定

   * 用户在照片中点选一条与机器 X 轴或 Y 轴平行的参考线；
   * 用户选择该线代表 X 方向还是 Y 方向；
   * 软件计算纸张坐标相对于机器坐标的旋转关系。

5. 单点物理归零流程提示

   * 软件不直接控制机器归零；
   * UI 明确提示用户：

     * 手动移动笔头到纸张左上角；
     * 在外部 sender 中执行 `G92 X0 Y0`；
     * 然后再执行本软件导出的 G-code。

6. SVG 纸面编辑器

   * 显示纸张；
   * 显示校正后的照片背景；
   * 显示标定点；
   * 显示机器参考线；
   * 支持放置测试路径；
   * 支持放置文本框对象；
   * 支持拖动对象；
   * 第一版不做单个笔画节点编辑。

7. 路径预览

   * 所有最终会写字的内容都应预览为路径；
   * 第一版可先用测试路径验证坐标，不必马上接入完整中文字库。

8. G-code 导出

   * 将路径对象转换为 G-code；
   * 支持配置：

     * 抬笔命令；
     * 落笔命令；
     * 绘制速度；
     * 空走速度；
     * Z 轴抬落笔速度；
     * Z 轴方向；
     * 机器工作区域；
     * 安全 Z；
     * 文件开头命令；
     * 文件结尾命令。
   * 只导出 `.gcode` 文件，不做串口发送。

9. 单点 Z 标定

   * 使用“点粗找 + 线细调”流程；
   * 支持用户自定义：

     * Z 轴正方向；
     * 粗找步长；
     * 细调步长；
     * 测试线数量；
     * 测试线长度；
     * 测试线间距；
     * 测试速度；
     * Z 轴移动速度；
     * 最大下降范围。
   * 第一版只做单点标定，多点只预留数据结构。

---

## 3. 暂不实现内容

以下内容不进入 MVP：

1. 串口发送器
   第一版只导出 G-code。用户使用已有的 Timesoft、LaserGRBL、UGS 或其他软件发送。

2. 自动 home
   当前机器没有限位开关，不执行 `$H` 等自动回零命令。

3. 真正自动调平
   没有压力传感器、触碰探针或闭环反馈，不做真正自动调平。

4. 多点 Z 补偿
   先预留数据结构，MVP 只实现单点 Z 标定。

5. 完整中文字库效果优化
   字库方案先用接口抽象，MVP 可先用测试路径；等坐标闭环跑通后再测试 Hanzi Writer / Make Me a Hanzi、Chinese Hershey Font 等方案。

6. 完整矢量编辑器
   不做类似 Inkscape 的路径节点编辑、贝塞尔控制点编辑、布尔运算、复杂图层系统。

7. Tauri 打包
   先用 Vite + React 开发网页原型。前端稳定后再考虑封装为 Tauri 桌面软件。

## 4. 技术路线

### 4.1 前端技术栈

推荐：

```text
Vite
React
TypeScript
SVG DOM
Zustand 或普通 React state
```

第一版优先使用 SVG DOM，而不是 Canvas。

原因：

* 本项目核心对象是矢量路径；
* SVG path 和 G-code 都是路径描述；
* SVG 元素可以被 React 直接管理；
* 方便调试路径、标定点、参考线；
* 方便后续导入 SVG 路径。

### 4.2 为什么使用 SVG DOM

SVG DOM 可以理解为浏览器中的矢量图元素树，例如：

```html
<svg>
  <path />
  <line />
  <rect />
  <g />
</svg>
```

每个路径、线段、文本框、标定点都可以作为一个元素或一组元素存在。

MVP 中：

* 纸张是 `<rect>`；
* 背景照片是 `<image>`；
* 标定点是 `<circle>`；
* 机器参考线是 `<line>`；
* 字体笔画和测试路径是 `<path>` 或 `<polyline>`；
* 一个文本框对象可以用 `<g>` 包住一组笔画路径。

## 5. 坐标系统设计

项目中至少有三套坐标系。

### 5.1 图片坐标系

照片原图中的像素坐标：

```ts
type ImagePoint = {
  x: number;
  y: number;
};
```

单位是 px。

用户点选纸张四角、机器参考线时，最初得到的都是图片坐标。

---

### 5.2 纸张坐标系

真实纸张上的毫米坐标：

```ts
type PaperPoint = {
  x: number;
  y: number;
};
```

推荐约定：

```text
纸张左上角 = (0, 0)
X 轴向右
Y 轴向下
单位：mm
```

例如 A4 纵向：

```text
左上角 = (0, 0)
右上角 = (210, 0)
右下角 = (210, 297)
左下角 = (0, 297)
```

---

### 5.3 机器坐标系

最终写入 G-code 的坐标：

```ts
type MachinePoint = {
  x: number;
  y: number;
};
```

机器坐标的原点由用户手动完成：

```gcode
G92 X0 Y0
```

也就是用户先手动把笔头移动到纸张左上角，然后让当前点成为机器工作坐标零点。

---

## 6. 坐标标定流程

### 6.1 纸张四角标定

用户在照片上点选四个角：

```ts
type PaperCornersPx = {
  topLeft: ImagePoint;
  topRight: ImagePoint;
  bottomRight: ImagePoint;
  bottomLeft: ImagePoint;
};
```

软件根据纸张真实尺寸建立从图片坐标到纸张坐标的变换。

需要提供核心函数：

```ts
function computeImageToPaperTransform(
  cornersPx: PaperCornersPx,
  paperWidthMm: number,
  paperHeightMm: number
): HomographyMatrix;
```

这个函数负责计算图片平面到标准纸面平面的透视变换矩阵。

---

### 6.2 机器参考线标定

用户在照片上点选两个点：

```ts
type MachineAxisLinePx = {
  p1: ImagePoint;
  p2: ImagePoint;
  axis: "x" | "y";
};
```

这条线代表机器 X 轴或 Y 轴方向。

软件先把这两个点从图片坐标转换到纸张坐标：

```ts
const p1Paper = imageToPaper(p1);
const p2Paper = imageToPaper(p2);
```

然后得到机器轴在纸张坐标系中的方向。

核心函数：

```ts
function computeMachineAxisInPaper(
  line: MachineAxisLinePx,
  imageToPaperMatrix: HomographyMatrix
): MachineAxisInPaper;
```

返回值：

```ts
type MachineAxisInPaper = {
  axis: "x" | "y";
  direction: Vector2;
  angleRad: number;
};
```

如果用户选择的是 X 轴参考线，则该方向就是机器 X 轴方向。
如果用户选择的是 Y 轴参考线，则需要旋转 90° 得到机器 X 轴方向。

---

### 6.3 纸张坐标到机器坐标

核心目标：

```text
用户在纸张坐标中放置的点，需要转换成机器 G-code 坐标。
```

核心函数：

```ts
function paperPointToMachinePoint(
  point: PaperPoint,
  calibration: CalibrationResult,
  machineConfig: MachineConfig
): MachinePoint;
```

基本逻辑：

1. 纸张左上角被认为是机器当前工作原点；
2. 机器 X 方向由机器参考线确定；
3. 机器 Y 方向由 X 方向垂直得到；
4. 纸张上的点投影到机器 X/Y 轴上；
5. 根据用户配置处理 Y 轴是否反向。

---

## 7. Z 标定流程

MVP 使用单点 Z 标定。

### 7.1 设计原则

Z 标定不是自动调平。
由于没有传感器，用户眼睛就是反馈源。

MVP 的目标不是找到绝对准确的纸面高度，而是快速找到：

```text
最轻但能连续写出清晰线条的落笔高度。
```

---

### 7.2 Z 标定配置

```ts
type ZCalibrationConfig = {
  mode: "none" | "single-point" | "multi-point";

  zPositiveDirection: "up" | "down";

  basePenDownZ?: number;

  coarseStepMm: number;
  fineStepMm: number;

  maxProbeDistanceMm: number;

  testLineCount: number;
  testLineLengthMm: number;
  testLineSpacingMm: number;

  testDrawFeedRate: number;
  zFeedRate: number;

  testArea?: {
    xMm: number;
    yMm: number;
    widthMm: number;
    heightMm: number;
  };

  points: ZCalibrationPoint[];
};
```

---

### 7.3 阶段 A：点粗找

用户在纸面上选择一个测试位置。

注意：
阶段 A 不需要每次平移。
它只用于快速判断“是否已经碰到纸”。
用户每点一次按钮，机器下降一个粗步长并在同一个位置点一下。
用户看到墨点后点击停止。

流程：

```text
1. 用户选择测试点；
2. 软件生成当前 Z 的点测试 G-code；
3. 用户执行；
4. 用户如果没看到墨点，点击“下降一步再点”；
5. 软件让 Z 再下降 coarseStepMm；
6. 再点一次；
7. 直到用户看到墨点，点击“看到墨点，进入细调”；
8. 如果超过 maxProbeDistanceMm 仍没有墨点，提示用户手动降低笔夹。
```

核心函数：

```ts
function generateCoarseDotProbeGcode(
  testPoint: PaperPoint,
  currentZ: number,
  config: ZCalibrationConfig,
  machineConfig: MachineConfig
): string;
```

按钮设计：

```text
下降一步并点一下
看到墨点，进入细调
全部没点上，提示调整笔夹
取消
```

阶段 A 的结果：

```ts
type CoarseProbeResult = {
  contactApproxZ: number;
};
```

这个高度不一定合适，可能偏重，也可能只是刚好有点。
它只用于缩小搜索范围。

---

### 7.4 阶段 B：线细调

阶段 B 在接触高度附近画少量短线。

一次只画 3～4 条，不要一次扫太多。

原因：

* 没写出来的线用户看不到；
* 线太多会污染纸面；
* 用户不容易判断编号；
* 本项目追求快速标定，不追求过度精密。

流程：

```text
1. 以 contactApproxZ 为中心；
2. 根据 fineStepMm 生成 3～4 个候选 Z；
3. 每个候选 Z 画一条短线；
4. 用户选择第一条“连续、清楚、无明显压痕”的线；
5. 保存该线对应的 Z 为 basePenDownZ。
```

核心函数：

```ts
function generateFineLineScanGcode(
  testArea: TestArea,
  contactApproxZ: number,
  config: ZCalibrationConfig,
  machineConfig: MachineConfig
): FineScanPlan;
```

返回：

```ts
type FineScanPlan = {
  candidates: {
    index: number;
    z: number;
    path: PolylinePath;
  }[];
  gcode: string;
};
```

用户选择结果：

```ts
type FineScanUserChoice =
  | { kind: "choose-line"; index: number }
  | { kind: "all-too-light" }
  | { kind: "all-too-heavy" };
```

如果全部太浅：

```text
整体向更深方向移动一组，再画 3～4 条。
```

如果全部太重：

```text
整体向更浅方向移动一组，再画 3～4 条。
```

最终保存：

```ts
basePenDownZ = selectedCandidate.z;
```

---

### 7.5 多点 Z 标定预留

MVP 不实现多点补偿，但数据结构要预留。

未来多点逻辑：

1. 第一个点完整执行：

   * 点粗找；
   * 线细调。
2. 其他点不重新粗找，只围绕 `basePenDownZ` 做局部 3 条线微调。
3. 多点结果可用于拟合平面或插值补偿。

预留类型：

```ts
type ZCalibrationPoint = {
  paperX: number;
  paperY: number;
  penDownZ: number;
};
```

## 9. 字体与路径设计

### 9.1 核心原则

普通字体 TTF/OTF 不是 MVP 主线。
普通字体是轮廓路径，写出来容易变成空心字。

本项目内部统一使用：

```text
stroke paths / polyline paths
```

也就是一组适合写字机直接走笔的路径。

---

### 9.2 字体接口

字体模块只负责：

```text
文本 → 笔画路径
```

定义接口：

```ts
interface StrokeFontProvider {
  id: string;
  name: string;

  supportsChar(char: string): boolean;

  getGlyph(char: string): Glyph | null;
}
```

字形类型：

```ts
type Glyph = {
  char: string;
  advanceWidth: number;
  strokes: GlyphStroke[];
};

type GlyphStroke = {
  points: PointMm[];
};
```

后续可以实现多个 provider：

```text
HanziWriterProvider
ChineseHersheyProvider
CustomStrokeFontProvider
```

MVP 可先只写接口和假字体 provider。
等坐标和 G-code 闭环跑通后再接入真实中文笔画数据。

---

### 9.3 文本对象

```ts
type TextObject = {
  id: string;
  type: "text";

  xMm: number;
  yMm: number;
  widthMm: number;
  heightMm?: number;

  rotationDeg: number;

  text: string;

  fontSource: string;
  fontSizeMm: number;
  letterSpacingMm: number;
  lineHeightMm: number;

  writingMode: "horizontal" | "vertical";
};
```

文本对象在项目文件中保存为文本和排版参数。
导出或预览时，再编译成路径。

function textObjectToPaths(

## 11. G-code 输出设计

### 11.1 G-code 生成输入

```ts
type GcodeJob = {
  paths: PolylinePath[];
  machineConfig: MachineConfig;
  zCalibration: ZCalibrationConfig;
};
```

### 11.2 核心函数

```ts
function generateGcode(job: GcodeJob): string;
```

### 11.3 输出逻辑

对每条路径：

```text
1. 抬笔；
2. 空走到路径起点；
3. 落笔；
4. 按路径点依次 G1 绘制；
5. 抬笔；
6. 处理下一条路径。
```

示例结构：

```gcode
G21
G90
; 用户应在运行前手动将笔头对齐纸张左上角，并执行 G92 X0 Y0

G0 Z5 F1000
G0 X10 Y20 F3000
G1 Z-1.2 F500
G1 X20 Y20 F1500
G1 X20 Y30 F1500
G0 Z5 F1000
```

### 11.4 安全检查

生成前必须检查：

```text
是否超出机器工作区
是否存在 NaN 坐标
路径是否为空
落笔 Z 是否未标定
纸张尺寸是否超过机器行程
参考线是否未标定
纸张四角是否未标定
```

核心函数：

```ts
function validateGcodeJob(job: GcodeJob): ValidationResult;
```

type CalibrationConfig = {

## 16. 第一版开发注意事项

1. 不要一开始接入复杂中文字库。
   先用测试路径验证坐标和 G-code。

2. 不要一开始做串口。
   先导出 G-code，用已有 sender 跑通。

3. 不要一开始做多点 Z。
   单点 Z 标定可用后，再扩展多点。

4. 不要把编辑器做成 Inkscape。
   MVP 只需要对象级拖动，不需要节点级路径编辑。

5. 不要假设机器有 home。
   当前所有位置基准都来自：

   * 照片标定；
   * 机器参考线；
   * 用户手动对齐纸张左上角；
   * G92 临时归零。

6. 所有速度、方向、Z 参数都必须可配置。
   不同机器的 Z 正负方向、Y 方向、抬落笔速度可能不同，不能写死。

## 17. MVP 完成定义

当满足以下条件时，MVP 视为完成：

```text
1. 可以导入纸张照片；
2. 可以点纸张四角；
3. 可以点机器参考线；
4. 可以在纸面上放置测试路径；
5. 可以导出 G-code；
6. 用户手动 G92 归零后，机器能把测试路径写到预期纸面位置；
7. 可以通过单点 Z 标定找到可用落笔高度；
8. 可以用假字体或简单 stroke 字体写出测试字符；
9. 所有关键机器参数可以在 UI 中配置；
10. 不依赖串口通信也能完成完整工作流。
```

MVP 的第一目标不是字写得多漂亮，而是：

```text
预览位置、纸张位置、机器输出位置三者能稳定对应。
```

只要这个闭环跑通，后续接入中文字体、多页批量、Z 多点补偿、Tauri、串口发送器都不会推翻主架构。
